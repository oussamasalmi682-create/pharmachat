1 Vue d'ensemble du projet

Problématique

Les pharmaciens reçoivent chaque jour des demandes de médicaments via WhatsApp et des applications mobiles. Gérer ces conversations en passant d'un écran à l'autre est fastidieux et source d'erreurs.

Solution : PharmaChat

PharmaChat est une interface web de style WhatsApp Web dédiée aux pharmaciens. Elle centralise toutes les conversations patients dans une seule application, avec :

  • Sidebar — liste de tous les patients avec indicateurs de non-lus
  • Zone de chat — historique complet des messages, envoi en temps réel
  • Actions rapides — réponses prédéfinies (Disponible, Rupture, Ordonnance…)
  • Temps réel — les nouveaux messages WhatsApp apparaissent immédiatement sans rechargement

4 Base de données — Prisma & MongoDB

Qu'est-ce que Prisma ?

Prisma est un ORM (Object-Relational Mapper). Il transforme le schéma déclaratif schema.prisma en un client TypeScript typé qui permet d'interroger MongoDB avec des fonctions simples au lieu d'écrire du SQL ou des requêtes Mongo brutes.

Schéma — prisma/schema.prisma

// Connexion à MongoDB
datasource db {
  provider = "mongodb"
  url      = env("DATABASE_URL")
}

// Modèle Patient
model Patient {
  id          String    @id @default(auto()) @map("_id") @db.ObjectId
  phone       String    @unique          // Numéro WhatsApp unique
  name        String
  channel     String    @default("whatsapp")
  status      String    @default("offline")
  lastSeen    String    @default("Hors ligne")
  unreadCount Int       @default(0)
  medications String[]                  // Tableau — avantage MongoDB
  avatarColor String    @default("#667781")
  messages    Message[]                 // Relation 1→N
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt
}

// Modèle Message
model Message {
  id          String   @id @default(auto()) @map("_id") @db.ObjectId
  content     String
  channel     String   @default("whatsapp")
  isSender    Boolean  @default(false)   // true = pharmacien, false = patient
  status      String   @default("delivered")
  whatsappId  String?                   // MessageSid Twilio (déduplication)
  patientId   String   @db.ObjectId
  patient     Patient  @relation(fields: [patientId], references: [id])
  createdAt   DateTime @default(now())
}

Relation Patient → Message

Un patient possède plusieurs messages (relation 1 → N). Chaque message contient patientId qui pointe vers le patient. Prisma gère cette jointure automatiquement via include: { messages: true }.

Connexion Prisma — lib/prisma.ts

En développement, Next.js recharge les modules fréquemment. Sans précaution, chaque rechargement créerait une nouvelle connexion MongoDB. Le pattern singleton global résout ce problème : 

import { PrismaClient } from '@prisma/client'

// Stocké dans globalThis pour survivre aux hot-reloads
const globalForPrisma = global as { prisma?: PrismaClient }

export const prisma = globalForPrisma.prisma ?? new PrismaClient()

if (process.env.NODE_ENV !== 'production')
  globalForPrisma.prisma = prisma

6 API REST — Routes Next.js

Next.js App Router permet de définir des routes API dans des fichiers route.ts. Chaque export (GET, POST…) correspond au verbe HTTP.

GET /api/patients/[id]/messages

Retourne l'historique des messages d'un patient depuis MongoDB.

export async function GET(_req, { params }) {
  const { id } = await params

  const rows = await prisma.message.findMany({
    where:   { patientId: id },
    orderBy: { createdAt: 'asc' },
  })

  return NextResponse.json(rows.map(toUIMessage))
}

POST /api/patients/[id]/messages

Envoie un message au patient. Fait 3 choses en séquence :

  1. 1Valide le contenu (non vide) et vérifie que le patient existe en base
  2. 2Sauvegarde le message dans MongoDB (isSender: true)
  3. 3Envoie le message WhatsApp via l'API Twilio si les variables d'env sont configurées
// Étape 2 : sauvegarde en base
const row = await prisma.message.create({
  data: {
    content,
    channel:  patient.channel,
    isSender: true,    // pharmacien envoie
    status:   'sent',
    patientId: id,
  },
})

// Étape 3 : envoi WhatsApp via Twilio
const client = twilio(accountSid, authToken)
await client.messages.create({
  from: `whatsapp:${from}`,
  to:   `whatsapp:${patient.phone}`,
  body: content,
})

POST /api/webhook — Réception WhatsApp

Twilio appelle cette URL automatiquement dès qu'un patient envoie un WhatsApp. C'est le point d'entrée de tous les messages entrants.

  1. 1Parse le corps URL-encodé envoyé par Twilio (From, Body, MessageSid)
  2. 2Déduplication : si le MessageSid existe déjà, on ignore (Twilio peut retenter)
  3. 3Upsert patient : crée le patient s'il est inconnu, sinon met à jour son statut
  4. 4Sauvegarde le message en base (isSender: false)
  5. 5Incrémente le compteur de messages non lus
  6. 6Émet via Socket.io pour mise à jour instantanée de l'UI
// Étape 6 : notification temps réel
const io = getIO()
if (io) {
  io.emit('new_message',   { patientId: patient.id, message: messageForUI })
  io.emit('unread_update', { patientId: patient.id, unreadCount: updated.unreadCount })
}

7 Temps réel — Socket.io

Pourquoi Socket.io ?

HTTP est unidirectionnel : le client demande, le serveur répond. Pour recevoir des messages sans recharger la page, il faut une connexion bidirectionnelle persistante — c'est ce que WebSocket (et Socket.io) fournit.

Contrainte Next.js

Next.js ne supporte pas nativement Socket.io dans son serveur interne. La solution : un serveur HTTP custom (server.js) qui encapsule Next.js et Socket.io sur le même port.

Serveur — server.js

const { createServer } = require('http')
const { Server }        = require('socket.io')
const next              = require('next')

const app        = next({ dev })
const httpServer = createServer(handle)   // Next.js gère les requêtes HTTP

const io = new Server(httpServer, {
  cors: { origin: '*' },
})

// Accessible depuis les routes API via globalThis.io
globalThis.io = io

io.on('connection', (socket) => {
  console.log('[Socket.io] Client connecté:', socket.id)
})

httpServer.listen(3000)

Accès depuis les routes API — lib/socket-server.ts

export function getIO(): Server | null {
  return globalThis.io ?? null
}

Client React — lib/socket-client.ts

Singleton pour éviter de créer plusieurs connexions WebSocket lors des re-renders React :

let socket: Socket | null = null

export function getSocket(): Socket {
  if (!socket) {
    socket = io({ path: '/socket.io', transports: ['websocket', 'polling'] })
  }
  return socket
}

9 Composants React

Server Components vs Client Components

Next.js App Router introduit deux types de composants. La règle est simple : tout est Server Component par défaut. On ajoute 'use client' uniquement si nécessaire.

Détail de chaque composant

Sidebar.tsx — Server Component

Charge la liste des patients depuis MongoDB et transmet à SidebarClient.

export default async function Sidebar() {
  const patients = await getPatients()  // requête MongoDB côté serveur
  return <SidebarClient patients={patients} />
}

SidebarClient.tsx — Client Component

  • Recherche : filtre par nom de patient ou médicament (useState)
  • Navigation active : surligne la conversation ouverte (usePathname)
  • Temps réel : écoute new_message et unread_update via Socket.io
  • Badges : icône WhatsApp/Mobile, compteur non-lus, aperçu dernier message

ChatContainer.tsx — Client Component

Le composant principal de conversation. Gère :

  • Affichage des messages avec séparateurs de dates ("Aujourd'hui", "Hier"…)
  • Bulles de chat : vert clair pour le pharmacien, gris pour le patient
  • Statut des messages : ✓ (envoyé), ✓✓ gris (reçu), ✓✓ bleu (lu)
  • Auto-scroll vers le dernier message
  • Envoi optimiste : le message apparaît immédiatement avant la réponse serveur
  • Socket.io : reçoit les nouveaux messages en temps réel

QuickActions.tsx — Client Component

10 Flux complet de données

Scénario A : Un patient envoie un message WhatsApp

  1. 1 Patient → WhatsApp → Twilio
    Le patient écrit "Bonjour, avez-vous du Doliprane ?" sur WhatsApp. Twilio reçoit ce message.
  2. 2 Twilio → POST /api/webhook
    Twilio envoie une requête HTTP POST à notre webhook avec les paramètres : From=whatsapp:+212601234567, Body=Bonjour, avez-vous du Doliprane ?, MessageSid=SM...
  3. 3 Webhook → MongoDB
    Le webhook crée ou met à jour le patient, sauvegarde le message, incrémente unreadCount.
  4. 4 Webhook → Socket.io → React
    Le webhook émet new_message et unread_update. ChatContainer et SidebarClient reçoivent ces événements et mettent à jour leur état React instantanément.
  5. 5 Affichage immédiat
    Sans rechargement : le message apparaît dans la conversation, et le badge rouge s'incrémente dans la sidebar.

Scénario B : Le pharmacien envoie une réponse

  1. 1 Saisie → Mise à jour optimiste
    Le pharmacien tape et appuie sur Entrée. ChatContainer ajoute immédiatement le message dans l'UI avec un ID temporaire (temp-1234), sans attendre le serveur.
  2. 2 POST /api/patients/[id]/messages
    En parallèle, une requête API est envoyée. Le serveur sauvegarde en MongoDB et appelle l'API Twilio.
  3. 3 Réponse → Remplacement
    À réception de la réponse 201, le message temporaire est remplacé par le message sauvegardé (avec le vrai ID MongoDB).
  4. 4 Twilio → Patient WhatsApp
    Twilio délivre le message sur le téléphone du patient via WhatsApp.

11 Concepts clés expliqués

App Router de Next.js

L'App Router remplace l'ancien Pages Router. Chaque dossier dans app/ devient une route URL. Le fichier page.tsx est le composant rendu. [patientId] entre crochets est un paramètre dynamique.

DossierURLRendu
app/page.tsx/Page d'accueil
app/chat/[patientId]/page.tsx/chat/6819abc...Conversation d'un patient
app/api/patients/route.ts/api/patientsAPI JSON

Mise à jour optimiste (Optimistic Update)

Technique UX qui affiche le résultat d'une action avant que le serveur confirme. Si le serveur échoue, on peut rollback. Ici, le message apparaît instantanément, puis est remplacé par la version confirmée.

Résultat perçu

L'expérience est identique à WhatsApp : le message s'affiche instantanément dès la frappe, sans latence réseau visible.

WebSocket vs HTTP polling

HTTP PollingWebSocket (Socket.io)
PrincipeLe client demande toutes les X secondesConnexion persistante bidirectionnelle
LatenceJusqu'à X secondesQuasi-instantanée (<50ms)
Charge serveurÉlevée (N clients × N requêtes/min)Faible (1 connexion par client)
Utilisé dans PharmaChatFallback automatiqueMode principal

Singleton Pattern

Utilisé deux fois dans ce projet :

  • PrismaClient — une seule instance de connexion DB pour toute l'application
  • Socket.io Client — une seule connexion WebSocket par onglet navigateur

Sans ce pattern, chaque re-render React ou rechargement Next.js créerait de nouvelles connexions, épuisant les ressources.

ORM (Object-Relational Mapper)

Prisma traduit des appels TypeScript en requêtes MongoDB. Avantages :

  • Autocomplétion complète dans l'IDE (prisma.patient.findMany({...}))
  • Types inférés automatiquement — pas d'erreurs de champ inexistant
  • Migrations de schéma gérées automatiquement (prisma db push)

Déduplication de messages

Twilio peut retenter l'envoi du webhook si la réponse tarde. Sans déduplication, le même message serait sauvegardé deux fois. La vérification sur whatsappId (MessageSid unique) évite ce doublon : 

const existing = await prisma.message.findFirst({ where: { whatsappId: messageSid } })
if (existing) return new NextResponse('', { status: 200 })  // Ignorer le doublon
